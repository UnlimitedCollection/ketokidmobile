# Parent App Design Plan

## Project
KetoKid Care - Parent Mobile App

## Goal
Build the parent-side mobile app flow for planning daily meals for a child on a keto program, based on hospital/admin-defined diet rules and food lists.

This document explains the exact UX flow, screen structure, behaviors, component states, and design consistency requirements so developers and AI agents can implement the app correctly.

---

## 1. Core Product Logic

### Parent flow
1. Parent logs in with hospital token
2. Parent sees Dashboard
3. Today's Meal Tasks cards are empty initially
4. Meal cards shown depend on keto type configured from admin side
5. Parent taps a meal card, for example Breakfast
6. Parent enters the Breakfast planning screen
7. Parent filters foods by KPI type:
   - Carb
   - Fat
   - Protein
   - Calories
8. Based on selected KPI, only matching foods are shown
9. Food list comes from admin-managed food master list
10. Parent can expand/collapse food cards
11. Parent can long-press food cards to select them
12. Selected cards become highlighted
13. Parent can select foods across multiple KPI groups
14. When there is at least one selected food, show `Save Today Plan` button
15. On save, return to Dashboard
16. Dashboard meal card now shows saved/filled state
17. Parent can copy meal to another meal, such as Lunch
18. Parent can edit saved meals later
19. Parent answers `Did child eat?`
20. If `Yes`, show portion/consumption UI
21. If `No`, mark meal as not involved / not eaten and do not count it as consumed food
22. Parent can also access History and Profile

---

## 2. Admin-Side Dependency

The parent app must not create the medical logic on its own.

### Admin controls
- Keto type
- Meal structure by keto type
- Food master list
- Food category mapping
- Which foods belong to:
  - Carb
  - Fat
  - Protein
  - Calories
- Child-specific meal planning availability

### Parent controls
- Select foods from allowed list
- Save meal plan for the day
- Copy meal to another meal
- Edit selected foods
- Mark whether child ate it
- Record portion consumed
- View history

## 3. Screen Flow

## 3.1 Login Screen
### Purpose
Secure parent access using token.

### UI
- App branding
- Token input field
- Continue button
- Help/support link

### Behavior
- Parent enters hospital-provided token
- On success, navigate to Dashboard

---

## 3.2 Dashboard Screen
### Purpose
Main home screen for the parent.

### Initial state
Today's Meal Tasks cards are empty.

### Content
- Greeting / child name
- Daily macro summary section
- Today's Meal Tasks
- Meal cards based on keto type
- View History button
- Profile access

### Meal cards
Examples:
- Breakfast
- Lunch
- Dinner
- Optional snack depending on keto type

### Card states
#### Empty state
- Meal has not been planned yet
- CTA: `Plan Meal`

#### Planned state
- Meal has selected foods saved
- Actions:
  - Edit
  - Copy to another meal
  - Did child eat? (Yes / No)

#### Consumed state
- Meal was eaten
- Show portion consumption summary

#### Not involved / not eaten state
- Parent selected `No`
- Meal should be visually marked as not eaten
- Food should not be counted as consumed for tracking

---

## 3.3 Meal Planning Screen
Example: Breakfast planning screen

### Purpose
Allow parent to select foods for that meal.

### Header
- Back button
- Meal title, such as `Breakfast Meal Plan`
- Support text, such as `Manage your child's daily nutrition`

### Top summary area
- Breakfast progress
- Date
- Daily progress indicator

### KPI filter row
The KPI cards act as category filters, not only as summary indicators.

#### KPI cards
- Carb
- Fat
- Protein
- Calories

### KPI behavior
- One KPI can be active at a time
- Active KPI is visually highlighted
- Food list updates based on active KPI
- Example:
  - If Carb is selected -> show only carb foods
  - If Fat is selected -> show only fat foods
  - If Protein is selected -> show only protein foods
  - If Calories is selected -> show calorie-related list if defined by admin mapping

---

## 3.4 Food List Section
### Source
Connected to admin food list.

### Food card behavior
Each card supports:
- Collapsed state
- Expanded state
- Selected state

### Collapsed card
Shows:
- Food image
- Food name
- Food type badge
- Quantity summary if applicable

### Expanded card
Shows:
- Food image
- Food name
- Description
- Quantity
- Any extra nutrition info needed
- Selection affordance

### Interaction
- Tap: expand/collapse card
- Long press: select card
- Multiple cards can be selected

### Selected state
- Card becomes highlighted
- Must match the style shown in design reference
- Highlight should be consistent across all KPI groups

### Important behavior
The parent is not limited to selecting only one KPI type.
They can:
- Select carb foods
- Then switch to fat
- Select fat foods
- Then switch to protein
- Select protein foods
- Then switch to calories if needed

Selections must persist when changing KPI filters.

---

## 3.5 Save Today Plan
### Visibility
Hidden until at least one food card is selected.

### When visible
Show a strong bottom CTA:
- `Save Today Plan`

### On click
- Save selected foods for the current meal and current day
- Return to Dashboard
- Update the selected meal card from empty state to planned state

---

## 3.6 Dashboard After Save
After saving a meal:
- Dashboard must show the meal as planned
- Meal card should support:
  - Edit foods again
  - Copy to Lunch / another meal
  - Did child eat? question

This is the state shown in the filled card design.

---

## 3.7 Copy Meal Feature
### Purpose
Speed up planning for similar meals.

### Example
- Parent plans Breakfast
- Then taps `Copy to Lunch`

### Behavior
- Copy selected food set into target meal
- Parent can still edit target meal afterward
- Show confirmation feedback after copy

---

## 3.8 Edit Meal Feature
### Purpose
Allow parent to reopen and modify the saved meal.

### Behavior
- Open same meal planning screen
- Restore previous selected foods
- Restore selected highlight states
- Parent can add/remove foods
- Save again to update plan

---

## 3.9 Did Child Eat? Interaction
This happens on Dashboard after the meal has been planned.

### Question
`Did child eat?`

### Options
- Yes
- No

### If Yes
Show consumption UI like the referenced design:
- Portion consumed options or equivalent
- Example:
  - 0%
  - 25%
  - 50%
  - 75%
  - 100%

### If No
- Remove consumed status from that meal
- Mark the meal as not involved / not eaten
- Do not count it toward consumption tracking
- Keep meal plan data if needed for edit/history, but not as eaten

---

## 3.10 History Screen
### Purpose
Allow parent to view previous meal records.

### Content
- Date-wise meal history
- Saved meals
- Eaten / not eaten state
- Portion consumed state
- Macro summary if available

---

## 3.11 Profile Screen
### Purpose
Parent account and child-related settings view.

### Content
- Parent profile
- Child information summary
- Token/account support
- Logout

## 4. UX Rules

## 4.1 Empty-to-filled progression
The flow must feel like this:
1. Empty meal card on dashboard
2. Enter meal planner
3. Select foods by KPI filters
4. Save Today Plan
5. Return to dashboard with filled meal card
6. Ask `Did child eat?`

This order is important.

---

## 4.2 Do not ask `Did child eat?` before meal planning
The `Did child eat?` question should appear in a meaningful way after the meal exists as a plan/saved meal.

---

## 4.3 Filter logic must not clear selections
When switching:
- Carb -> Fat
- Fat -> Protein
- Protein -> Calories

Previously selected foods must remain selected.

---

## 4.4 Long-press selection is required
Food selection must use long-press interaction as specified by the design direction.
Optional improvement:
- Also support tap-to-open and long-press-to-select
- Add subtle onboarding hint if needed

---

## 4.5 Expand/collapse must remain available
A selected card should still be expandable/collapsible.

---

## 4.6 Save CTA behavior
- Hidden when nothing is selected
- Visible when one or more foods are selected
- Sticky bottom positioning recommended

---

## 6. Design System Rules

## 6.1 Visual alignment
All screens must stay aligned with the provided design references.

### Must remain consistent
- Color palette
- Rounded card style
- Soft shadows
- Green primary tone
- Blue secondary KPI tone
- Clean white surfaces
- Large mobile-friendly spacing
- Soft medical/family-safe look

---

## 6.2 Color usage
### Primary
- Green for main action, positive states, progress
- Used for:
  - Continue
  - Active save actions
  - Selected positive toggles
  - Progress sections

### Secondary
- Blue for carb KPI and filter emphasis

### Tertiary / accent
- Warm tone for protein or supporting accents

### Neutral
- White / light gray backgrounds
- Soft dark text

### Selection highlight
- Must match selected card style shown in design reference
- Keep it clearly visible but not harsh

---

## 6.3 Component consistency
Use the same style for:
- Buttons
- KPI cards
- Food cards
- Dashboard meal cards
- Toggle pills
- Bottom CTA bars

Do not redesign screen-by-screen in different visual languages.

---

## 7. Developer State Logic

## 7.1 Dashboard meal card state machine
### Empty
- No foods selected for today/meal

### Planned
- Foods saved
- Show edit + copy + did child eat

### Ate = Yes
- Show portion selection
- Track consumed state

### Ate = No
- Mark as not involved / not eaten
- Exclude from consumed metrics

---

## 7.2 Meal planner selection state
For each food card:
- default
- expanded
- selected
- selected + expanded

---

## 7.3 KPI state
For KPI cards:
- inactive
- active

Only one KPI card active at a time for filtering.

## 9. AI Agent Implementation Notes
If an AI coding agent reads this file, it should follow these rules:

1. Preserve the provided UI direction
2. Do not invent new meal logic outside admin control
3. Keep dashboard cards empty first, then filled after save
4. Use KPI cards as active food filters
5. Keep food selections persistent across filters
6. Support expand/collapse and long-press select
7. Show `Save Today Plan` only when selection exists
8. After save, return to dashboard updated state
9. Allow copy and edit
10. Ask `Did child eat?`
11. If `Yes`, show portion UI
12. If `No`, mark as not involved and exclude from consumption tracking
13. Keep colors, spacing, and component styling aligned across screens
14. Include History and Profile screens

## 11. Final Summary

This parent app is a guided meal-planning and meal-tracking system.

The correct experience is:

- login with token
- see empty dashboard meal cards
- open a meal
- filter foods using KPI cards
- select foods from admin-managed food list
- save today plan
- return to dashboard with filled meal state
- copy/edit if needed
- answer did child eat
- if yes, capture portion
- if no, mark not involved
- access history and profile

All screens must remain visually aligned with the current design direction and color system.
