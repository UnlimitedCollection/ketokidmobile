# Role Change Plan: Admin and Moderator

## 1. Overview

The system will now have **two roles only**:

1. **Admin**
   - Has full access to the system
   - Can view, create, edit, update, and delete data
   - Can manage tokens and system settings

2. **Moderator**
   - Has read-only access
   - Can view all data in the system
   - Cannot create, edit, update, or delete anything

---

## 2. Role Definitions

### Admin
Admin has complete control over all modules in the platform.

Admin can:
- View dashboard
- Manage doctors
- Manage kids
- Manage foods
- Manage recipes
- Manage meals
- Manage tokens
- Update system settings
- Perform all create, edit, update, and delete actions

### Moderator
Moderator is a view-only user.

Moderator can:
- View dashboard
- View doctors
- View kids
- View foods
- View recipes
- View meals
- View token status
- View reports and analytics

Moderator cannot:
- Add new records
- Edit existing records
- Delete records
- Generate or reset tokens
- Change system settings

---

## 3. Permission Model

Use **Role-Based Access Control (RBAC)**.

### Example Role Setup

```json

  "moderator": ["read_only"]
```

## 5. Backend Plan

### 5.1 Add Role Field to User Table

Example:

```sql
users (
  id,
  name,
  email,
  password,
  role
)
```

Possible values for `role`:
- `admin`
- `moderator`

### 5.2 API Permission Rules

| HTTP Method | Admin | Moderator |
|------------|------|-----------|
| GET | Allowed | Allowed |
| POST | Allowed | Not allowed |
| PUT / PATCH | Allowed | Not allowed |
| DELETE | Allowed | Not allowed |

### 5.3 Middleware Example

#### Admin-only middleware

```js
function authorizeAdmin(req, res, next) {
  if (req.user.role !== "admin") {
    return res.status(403).json({ message: "Forbidden" });
  }
  next();
}
```

#### Read-only restriction for moderator

```js
function restrictModeratorWrite(req, res, next) {
  if (req.user.role === "moderator" && req.method !== "GET") {
    return res.status(403).json({ message: "Read-only access" });
  }
  next();
}
```

## 6. Frontend Plan

### 6.1 Admin UI
Show all action buttons:
- Add
- Edit
- Delete
- Save
- Update
- Reset token
- Generate token

### 6.2 Moderator UI
Show only:
- View pages
- Tables
- Reports
- Dashboard
- Detail pages

Hide or disable:
- Add buttons
- Edit buttons
- Delete buttons
- Save buttons
- Token action buttons
- Settings update actions

### 6.3 Read-only Indicators
To make the moderator role clear in UI:
- Show a role badge: **Moderator (View Only)**
- Disable form fields if they appear
- Hide action sections where possible

## 7. Security Rules

- Moderator must never be able to call create, update, or delete APIs successfully

## 12. Role Management (Admin Controls Roles)

### Admin can manage Admins and Moderators:

Admin can:
- Create new Admin accounts
- Create new Moderator accounts
- Edit Admin and Moderator details
- Delete Admin and Moderator accounts
- Assign roles (Admin ↔ Moderator)
- Change roles of existing users

### Role Assignment Rules:
- Only Admin can assign or change roles
- Moderator cannot change roles
- Admin can promote Moderator → Admin
- Admin can demote Admin → Moderator

👉 This ensures full control over system access and permissions

---

## 13. Security Considerations for Role Management

- Only Admin endpoints allow role changes
- Role change actions must be logged (audit log recommended)
- Prevent self-demotion if it leaves system without Admin
- Optional: require confirmation for role changes
